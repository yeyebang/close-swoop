## 核心目标
- 每日14:00扫描A股，找15:00前涨停票，次日卖出
- 自主爬取数据，本机运行，Python编写

## 技术路线
- AKShare (免费A股数据) → 规则筛选 → 代理回测 → 监督学习排序 → 候选名单
- 运行: `python run.py scan` (可定时14:00)
- Web工作台: `python run.py ui`，默认访问 `http://127.0.0.1:3002`

## 关键约束
- A股T+1: 今日买明日卖
- 涨停: 主板+10%, 创业板/科创板+20%, ST+5%
- 涨停概率极低(~<1%), 目标是提高胜率

## 功能模块
1. 数据采集 - AKShare拉全市场行情 + 历史日K
2. 特征工程 - 量比/换手/均线/涨幅/价格位置
3. 代理回测 - 使用缓存日线先验证规则是否有统计优势
4. 信号扫描 - 规则预筛 + ML排序
5. 大模型辅助 - Ollama用于解释候选股，不作为核心预测器
6. 通知推送 - 终端输出 + 可选微信推送

## 旧聊天关键结论
- 输出目标不是泛泛分析，而是每日给出最优的几个股票代码，先做虚拟盘，虚拟盘命中稳定后再考虑实盘。
- 当前模型应当“每天跑、次日复盘、累计样本后调整排序分”，不是只靠一套固定规则打分。
- 自适应评分需要已复盘样本达到阈值才启用，默认 `paper.adaptive_min_samples = 30`，避免样本太少时过拟合。
- 利好/题材信息会影响尾盘涨停概率，但应作为增强因子，不应替代行情、成交、盘口等核心量化数据。
- 当前本地网络或沙盒环境可能出现财经站 DNS 失败；代码设计上允许接口失败后回退到缓存，但缓存只适合应急参考。

## 已完成的重要改动
- 实时行情主链路: 东方财富优先，新浪备用，全部失败才使用旧缓存。
- 缓存机制: `realtime_cache_ttl_seconds` 默认 120 秒；结果中写入 `数据源` 和 `行情采集时间`，避免误把旧行情当实时行情。
- 隔天缓存 bug 已修: 昨天的实时缓存过期时不会再因为 `cache_age=None` 崩溃。
- 虚拟盘闭环: `scan` 会复盘未结算推荐、写入本次 Top 候选，并在样本足够后启用自适应排序。
- 二次增强: 规则/自适应先筛出候选，再对 Top 候选做同花顺榜单/概念和腾讯分笔验证。
- 增强字段: `enrich_score`、`enrich_notes` 会写入结果和虚拟盘。
- Web UI: 已有本地工作台，可查看汇总、结果、历史、虚拟盘，并可从 UI 启动扫描/Ollama 分析。
- UI 进度反馈: 扫描和 Ollama 分析都有弹窗、运行计时、阶段提示、实时日志和后台运行能力。

## 数据源分层
- 第一层全市场扫描: 东方财富 > 新浪 > 旧缓存应急。
- 第二层候选股验证: 腾讯分笔、东方财富盘口、雪球个股等适合只对 Top 候选做补充。
- 第三层题材因子: 同花顺概念板块、同花顺技术榜、东方财富板块。
- 第四层新闻公告: 巨潮资讯、东方财富资讯、同花顺题材等，后续可作为利好/风险因子接入。

## 常用命令
- 实时扫描: `python run.py scan`
- 定时扫描: `python run.py --auto`
- 日线代理回测: `python run.py backtest`
- 训练模型: `python run.py train`
- 输出最新缓存候选: `python run.py candidates 20`
- 查看虚拟盘复盘报告: `python run.py paper-report`
- Ollama解释候选股: `python run.py explain qwen3.6:35b-a3b-q8_0`
- 打开本地Web工作台: `python run.py ui`

## 当前注意事项
